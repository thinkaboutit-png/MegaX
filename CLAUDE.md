# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Live Klipper configuration for a heavily upgraded **Anycubic Mega X** (Trigorilla board, atmega2560). This directory is `/home/mark/printer_data/config` on the printer host **harrytang.local** (hostname `HarryTang`), and Claude Code runs natively on that host — these are local files, with direct access to the services, logs, and systemd units.

This is the **default instance**: services `klipper` / `moonraker` (unsuffixed), Moonraker on port **7125**. The host also runs Ender3v2 and PulseXE instances — see `/home/mark/CLAUDE.md`. Don't touch those from here.

## Talking to the printer

Local shell and the Moonraker API on localhost both work:

```bash
curl -s http://localhost:7125/printer/info                       # state: "ready", "error", ...
curl -s http://localhost:7125/printer/objects/query?print_stats  # is a print running
```

## Restart policy

**Never restart Klipper or Moonraker while an operation is in progress.** An operation means a print running or paused mid-print, or the printer homing, probing, meshing, calibrating, or heating toward a target. Restarting mid-operation aborts the job and can leave the hotend parked in the part.

Check before every restart, and treat anything other than an idle/complete state as in progress:

```bash
curl -s http://localhost:7125/printer/objects/query?print_stats
```

When idle, apply config changes with `curl -X POST http://localhost:7125/printer/restart` (or `firmware_restart` after `[mcu]`/pin-level changes), then confirm `printer/info` reports `ready` and check `../logs/klippy.log` for config errors. If a print is active, say a restart is needed and stop — do not queue it, and don't send state-changing G-code either.

## Git workflow

- Commit after a change is verified (user restarts, Klipper reports "ready"), one commit per logical change. Don't commit unverified config edits.
- Config comments use dated notes, e.g. `#05-17-26: Test tower says 1500` or `#03/18/26 - MDR: Added for tailscale access`. Follow that style when changing values — record the date and why.

## Config layout

- `printer.cfg` — root config. The `SAVE_CONFIG` block at the bottom (`#*#` lines) is auto-generated by Klipper; never hand-edit it. PID values live there, so the `control:/pid_*` lines above are intentionally commented out.
- `macros/start_print.cfg`, `macros/end_print.cfg` — the live START_PRINT/END_PRINT macros (included from printer.cfg). START_PRINT has Z_TILT_ADJUST and BED_MESH_CALIBRATE steps commented out because no probe is installed yet — bed leveling is manual via `[bed_screws]`. Don't re-enable them.
- `macros/misc.cfg` — **not included** in printer.cfg. It's a reference copy of Klipper's example macros and contains an older START_PRINT; don't confuse it with the live one.
- `mainsail.cfg` → `/home/mark/mainsail-config/mainsail.cfg` and `timelapse.cfg` → `/home/mark/moonraker-timelapse/klipper_macro/timelapse.cfg` are symlinks into upstream repos managed by Moonraker's `update_manager`. They resolve correctly; don't "fix" or replace them with copies, or upstream updates stop reaching this instance.
- `printer-YYYYMMDD_HHMMSS.cfg` — automatic backups created on SAVE_CONFIG. Read-only history; don't edit.
- `hardware/` and `tuning/` — empty placeholders for a planned split of printer.cfg: per-device sections (TMC2209, probe, runout sensor) into `hardware/`, calibration configs into `tuning/`.
- `moonraker.conf`, `KlipperScreen.conf`, `crowsnest.conf`, `sonar.conf` — service configs. KlipperScreen.conf has its own auto-generated `#~#` block at the bottom.

## Hardware state (as of July 2026)

- X/Y/Z/Z1 run BTT TMC2209s in **standalone mode** (MS1/MS2 set in hardware). There are deliberately no `[tmc2209 ...]` sections in the config — don't add them.
- Dual Z motors, each with its own endstop (`stepper_z1` has `endstop_pin`).
- Extruder is a Bondtech LGX (`rotation_distance: 7.95`) on the **stock driver**, Capricorn bowden. Planned: TMC2209 for the extruder, MK4-style carriage / direct-drive conversion, hotend upgrade, replacing the loose stock bowden coupler (Bondtech LGX Bowden Interface Plug).
- Filament runout sensor: pin PC13 confirmed on the board; the commented-out `[filament_switch_sensor]` in printer.cfg still references `^!PJ1` — reconcile before enabling.
- Hotend thermistor is suspected to be failing (see the `min_extrude_temp` note in printer.cfg); treat odd temperature readings as likely hardware, not config.
- Input shaper values came from tuning-tower calibration (no accelerometer), Y skips steps around 3000 mm/s² — hence `max_accel: 1500`.

## Slicer

Cura. Its machine start G-code calls `START_PRINT BED_TEMP={material_bed_temperature_layer_0} EXTRUDER_TEMP={material_print_temperature_layer_0}` and end G-code calls `END_PRINT`, so temps and sequencing belong in the macros, not in Cura.
